# 互动区处理规则（唯一执行标准）

## 1. 目标

确保互动区每条候选人消息都有一致处理结果，避免“扫到了但没回复”。

## 2. 五种固定场景与标准话术

### 场景 1：首次打招呼

您好，我们是一家高新信息技术集团，目前岗位地址在：鞍山、苏州、达州、北京、长春，看到您有最近均从事该相关工作，请问是否有意向了解一下我公司。

### 场景 2：候选人有回复但未发简历（默认场景）

请问能否留下您的简历，以方便我们就您的简历匹配度进一步沟通？

规则：只要候选人已回复且未发简历，默认走场景 2。

### 场景 3：候选人明确拒绝职位

好的，如果您这边随时需要了解其他空缺岗位，可以交换微信，随时联系我。

### 场景 4：候选人已发简历

感谢您对公司的认可，我们已经收到您的简历，后续会有人事专员与您对接，请保持您的手机畅通。

识别信号包括两类：

- 附件简历组件（如“查看附件简历”）
- 在线简历提示/链接（如“已发送在线简历”“在线简历”超链接）

### 场景 5：候选人发简历后再次发消息

先回复：请稍等  
然后转到人工处理。

## 3. 核心业务边界

1) 未收到简历前，不转人工（默认走场景 2；若明确拒绝走场景 3）。  
2) 仅在“已发简历”相关场景进入人工链路（场景 4、场景 5）。  
3) 每次扫描到可处理会话都要给反馈，不允许静默跳过。

## 4. 动作映射

- `resume_request`：未收到简历，一律索要简历。  
- `closing`：候选人明确拒绝，礼貌结束。  
- `ack_and_handover`：检测到附件简历，先确认收到，再转人工。  
- `handover`：仅允许在“已收到简历后继续追问”触发。  
- `noop`：仅用于重复消息去重或无需重复动作场景。

## 5. 会话状态流转

- 初次打招呼后：`contacted`  
- 索要简历后：`resume_requested`（或等待中的 `awaiting_reply`）  
- 收到简历后：`resume_received`  
- 收简历后继续追问：`needs_human_takeover`  
- 明确拒绝：`not_interested_reasoned`

## 6. 兜底扫描规则

- 先扫未读；未读为 0 时再扫最近会话。  
- 最近会话中若命中“未发简历”，默认仍执行索要简历。  
- 最近会话中若命中“已发简历”，执行“确认收到 + 转人工”。

## 7. 去重与安全

- 同一会话 + 同一候选人 + 同一最新候选人消息，只处理一次。  
- 已处理过的消息写入去重键，防止重复回复。  
- `dryRun=true` 时只记录动作，不实际发送。
- 每次互动处理都会把该会话的原始对话、判定动作、回复文案写入本地审计日志（JSONL）。
- 每轮任务还会写入全局时间线日志（开始/事件/结束/异常），用于按时间顺序复盘。

## 8. 当前扫描上限

- 每轮扫描上限由 `config/recruit-agent.json` 的 `interaction.unreadLimit` 控制。  
- 当前值：`20`。
- 会话审计日志目录由 `storage.interactionLogDir` 控制，默认 `data/interaction-logs`。
- 动作时间线目录由 `storage.actionLogDir` 控制，默认 `data/action-logs`。

## 9. 文案标准

- 场景文案必须按本文件固定文本执行。  
- 不得自由发挥，不得额外承诺薪资、offer、入职日期。  
- 场景 5 必须先发“请稍等”，再转人工。
- 互动发送前必须加入随机人类化间隔（使用 `guardrails.minDelayMs ~ maxDelayMs`）。
- 标签切换、会话点击、打招呼按钮点击等 UI 点击动作也要加入人类化间隔（避免连续机械点击）。

## 10. 生效优先级

本文件优先于其他散落描述；如与其他文档冲突，以本文件为准并同步修订。
