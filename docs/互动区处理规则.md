# 互动区处理规则

## 一、扫描策略（单遍处理）

Agent 每次进入互动区，按以下两个阶段执行，**每个会话只点一次，点进去立刻判断并回复，处理完再进入下一个**。

### 阶段一：优先处理未读消息

1. 逐条检查会话列表，发现红色未读角标（数字≥1）
2. 立即点进该会话
3. 读取消息内容 → 判断场景 → 立即发送回复
4. 处理完毕后继续向下寻找下一个未读会话
5. 直到列表中无更多未读为止

### 阶段二：无未读时的兜底扫描

仅当阶段一**未发现任何未读**时触发：

1. 从列表顶部开始，最多处理 20 个会话
2. **预过滤**：列表预览文字若包含我方话术前缀，直接跳过（不点进去）
3. 对其余会话：点进去 → 读消息 → 判断 → 立即回复（如需）→ 处理完再看下一个

---

## 二、五种对话场景与话术

| 场景 | 触发条件 | 回复话术 |
|------|----------|----------|
| **场景1：首次打招呼** | 推荐/搜索阶段主动发出 | "您好，我们是一家高新信息技术集团，目前岗位地址在：鞍山、苏州、达州、北京、长春，看到您有最近均从事该相关工作，请问是否有意向了解一下我公司。" |
| **场景2：有回复但未发简历** | 候选人回了任何内容，但未检测到简历 | "请问能否留下您的简历，以方便我们就您的简历匹配度进一步沟通？" |
| **场景3：明确拒绝** | 判断为 negative（明确表示不感兴趣） | "好的，如果您这边随时需要了解其他空缺岗位，可以交换微信，随时联系我。" |
| **场景4：候选人发了简历** | 检测到附件简历卡片或在线简历链接/文字 | "感谢您对公司的认可，我们已经收到您的简历，后续会有人事专员与您对接，请保持您的手机畅通。" |
| **场景5：发简历后继续追问** | 状态为 resume_received，候选人再次发消息 | 先回复"请稍等"，然后标记转人工 |

> **优先级**：场景5 > 场景4 > 场景3 > 场景2（场景2是兜底，绝大多数未发简历的情况都用场景2）

---

## 三、简历识别规则

以下任意一种情况判定为"已发简历"：

- 候选人消息气泡中包含 **附件简历卡**（含文字"查看附件简历"或"附件简历"）
- 候选人消息气泡中包含 **在线简历链接**（`<a>` 标签文字含"在线简历"）
- 候选人消息气泡中包含文字：**"在线简历"、"已发送在线简历"、"发送在线简历"、"发送了在线简历"**

---

## 四、跳过规则（不发任何回复）

以下情况直接跳过当前会话：

1. **我方最后发言**：会话最后一条消息与我方话术模板匹配（防止自言自语）
2. **已处理过**：该消息已在本轮或历史轮次中处理并记录（去重 key = `候选人ID + 消息前120字`）
3. **候选人已彻底关闭**：状态为 `not_interested_reasoned`，不重复回复

---

## 五、转人工规则

以下情况进入人工接管队列（记录到 `data/recruit-agent-state.json` 的 `manualHandovers`）：

- 候选人发了简历（场景4完成后）
- 候选人发简历后继续追问（场景5）
- 到期跟进但找不到对应会话

---

## 六、到期跟进

对于状态为 `contacted / resume_requested / awaiting_reply` 且跟进时间已到的候选人，在主扫描结束后单独处理：重新定位其会话，发送一次跟进消息。

---

## 七、发送延迟

每次发送消息前，随机等待 **3~8 秒**（模拟人类操作节奏，避免被平台识别为机器人）。

---

## 八、配置参数

| 参数 | 位置 | 说明 |
|------|------|------|
| `interaction.unreadLimit` | `config/recruit-agent.json` | 阶段一最多扫多少条未读（默认20） |
| `guardrails.minDelayMs` | `config/recruit-agent.json` | 发送前最小等待毫秒（默认3000） |
| `guardrails.maxDelayMs` | `config/recruit-agent.json` | 发送前最大等待毫秒（默认8000） |
| `daemon.interactionIntervalMinutes` | `config/recruit-agent.json` | 守护进程轮询间隔（默认1分钟） |
