# 招聘 Agent 需求文档（持续更新）

## 1. 文档定位

本文件是招聘 Agent 的产品 + 开发统一需求基线。  
目标是保证：需求、实现、验证、使用手册始终同频。

## 2. 业务目标

- 24 小时不间断值守招聘流程。
- 1 分钟粒度处理互动消息，尽量控制候选人等待时间。
- 自动完成高重复动作：筛选、首轮触达、索要简历、状态记录、转人工。
- 对敏感和不确定场景优先安全，避免误回。

## 3. 核心流程

1. 同步职位中心岗位。  
2. 推荐/搜索/潜在页面发现候选人并评分。  
3. 达阈值后发送首轮招呼（不直接索简历）。  
4. 互动页读取完整历史对话并做决策：  
   - 候选人积极回复 -> 委婉索简历  
   - 明确不感兴趣 -> 礼貌结束  
   - 识别到附件简历组件 -> 致谢并转人工  
   - 未收到简历前（含敏感/未知问题）-> 统一先索要简历  
5. 全量记录状态、日志、人工接管清单。

## 4. 非功能需求

- 稳定性：单任务失败不拖垮守护进程。
- 可追溯：每次决策可回放（输入摘要、模型输出、执行结果）。
- 安全性：动作白名单、频控、黑名单、敏感词兜底。
- 可维护：配置驱动 + 模块化，避免硬编码分散。

## 4.1 AI 原生决策原则

- 能由 AI 基于上下文稳定判断的场景，不再以关键词规则作为主链路。
- 规则系统只承担安全边界与异常回退，不与 AI 主链路竞争决策权。
- 所有自动动作必须由结构化决策输出驱动，禁止散落式 if-else 关键词决策。

## 5. LLM 决策需求（互动页）

### 5.1 输入

- 双方最近 N 轮完整消息（默认 6~12 条，按实现可调）。
- 候选人状态与岗位上下文。
- 是否出现“查看附件简历/附件简历”组件。
- 风控规则提示（敏感词、拒绝提示词、积极提示词）。

### 5.2 输出

固定 JSON：

- `intent`: `positive | negative | resume_sent | question | sensitive | unknown`
- `action`: `resume_request | closing | ack_and_handover | handover | noop`
- `reply_text`: 文案
- `confidence`: 0~1
- `reason`: 简述依据

### 5.3 执行规则

- 执行层只接受白名单 action。
- `ack_and_handover` 后必须进入人工跟进链路。
- `handover` 仅在已收到简历后才允许触发。
- 重复消息去重，避免同一上下文重复触发。

## 6. 当前版本边界

- 仅支持智联招聘页面自动化（Playwright）。
- 推荐/潜在页默认处理首个激活岗位，避免错岗归属。
- 简历附件当前保存“线索和对话证据”，后续可扩展下载文件本体。

## 7. 迭代清单（按优先级）

- P0：LLM 决策接入（互动页）+ 规则兜底双轨。
- P0：决策日志标准化（便于复盘）。
- P1：失败重试与会话定位鲁棒性提升。
- P1：可配置的模板分层（岗位/场景）。
- P2：运营看板（接管量、回复时效、触达转化）。

## 8. 验收标准

- 能连续运行 24h，互动轮询维持 1 分钟。
- 未读互动可在下一轮内处理，不遗漏明显会话。
- 对“拒绝/已发简历/未发简历索要”三类场景处理正确率可人工抽检通过。
- 状态文件、日报、人工接管列表一致。

## 11. 当前对齐需求（2026-03-27）

### 11.1 目标（第一性原理）

- 核心输入：岗位关键字 + 地域约束（东北）。
- 核心动作：搜索候选人 -> 触达候选人（打招呼/可发首条消息时索要简历）。
- 核心产出：候选人回复后进入互动区，最终拿到并下载简历，沉淀到本地简历知识库。

### 11.2 业务闭环（最小可行）

1. 在人才搜索页使用岗位关键字检索候选人（东北地区优先）。  
2. 每次任务最多触达 20 个候选人（风控硬约束）。  
3. 触达阶段先打招呼；若页面能力允许，直接发送礼貌索简历话术。  
4. 触达完成后结束前链路，不在该阶段做复杂多轮对话。  
5. 候选人有回复后，由互动区既有逻辑接管处理。  
6. 候选人发送简历后，执行下载与归档，进入简历资产沉淀链路。  

### 11.3 当前已实现（基于现有代码/文档确认）

- 已有搜索能力（推荐/搜索/潜在与互动流程均已实现）。
- 已有互动区处理能力（索要简历、拒绝结束、简历致谢、转人工）。
- 已有互动区简历下载能力（附件/在线简历信号识别 + 下载逻辑）。
- 已有简历归档命名与证据保存能力（本地文件 + 结构化 JSON 线索）。
- 已有会话去重与重复回复防护（interaction handled key + thread key）。
- 已有单日触达上限能力，默认 `dailyContactLimit = 20`。

### 11.4 待实现（本轮需求新增）

- 平台对齐：以“智联招聘”为唯一目标平台（与当前代码主平台一致）。
- 前链路串联：在“岗位关键字搜索”阶段显式加入“打招呼后立即索简历（若可发消息）”的确定性动作。
- 能力探测：登录实测“打招呼”和“发送消息”是否可连续自动执行，并将结果配置化。
- 地域过滤增强：前链路按“东北地区”做硬过滤（城市映射表需配置化）。
- 批次控制增强：明确“每次任务最多 20 人”的任务级上限，而不只依赖日上限。

### 11.5 非目标（本轮不做）

- 不新增复杂对话策略（继续复用互动区现有能力）。
- 不做大规模模型策略重构（保持当前规则/LLM架构）。
- 不扩展新的报表系统或管理后台。

### 11.6 验收标准（本轮）

- 可基于岗位关键字稳定检索东北候选人列表。
- 单次任务最多触达 20 人，且触达行为可审计。
- 若平台允许，触达后首条消息可自动索要简历；若不允许，保底打招呼成功。
- 候选人回复后，能无缝进入互动区既有链路处理。
- 简历下载成功率可稳定复现，且本地归档可追溯（候选人/会话维度）。

### 11.7 搜索人才打招呼：已遇到问题与实现结论（2026-03-28）

| 现象 | 原因 | 处理 |
|------|------|------|
| 启动报 `SingletonLock` / profile 已占用 | 上次浏览器未关或异常退出残留锁文件 | 启动前自动结束占用 `SingletonLock` 的进程并清理锁文件后重试 |
| 只打了约 5～6 人而非 20 | 首屏 DOM 仅挂载少量 `.search-resume-item-wrap`，懒加载未触发 | 搜索后向下滚动列表，直至行数达到 `maxCandidatesPerQuery` 或不再增加 |
| 解析 0 条或只处理 1 人 | 曾受 `topContactCount`、列表层 `excludeKeywords` 过滤 | `topContactCount` 为 0 时按 `maxCandidatesPerQuery` 拉满；搜索列表不再按岗位排除词过滤 |
| 「未聊过」日志提示未找到 | 部分页面为自定义筛选项，非标准 `checkbox` | 增加文案「未聊过」检测；筛选仍以页面「未聊过」为准 |
| 选职弹窗后长时间无操作 | 原固定 `sleep(3500)` 等 IM | 改为轮询直至出现聊天输入框（或超时），点确定后等待弹窗关闭 |
| 两人间隔过长 | `guardrails.minDelayMs/maxDelayMs` 过大 | 可按真人节奏改为约 1000～2000ms |
| 日志「第 N 套定位器」 | KM 职位搜索框多套 Playwright 定位器依次尝试 | 仅调试信息，表示第 N 套选择器命中输入框 |
