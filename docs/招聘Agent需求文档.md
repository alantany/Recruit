# 招聘 Agent 需求文档（持续更新）

## 1. 文档定位

本文件是招聘 Agent 的产品 + 开发统一需求基线。  
目标是保证：需求、实现、验证、使用手册始终同频。

## 2. 业务目标

- 24 小时不间断值守招聘流程。
- 1 分钟粒度处理互动消息，尽量控制候选人等待时间。
- 自动完成高重复动作：筛选、首轮触达、索要简历、状态记录、转人工。
- 对敏感和不确定场景优先安全，避免误回。

### 2.1 产品功能清单（智联招聘 · 与当前实现对齐）

以下为 **已实现、可配置** 的能力，供产品对外说明与验收对齐。

| 模块 | 能力说明 |
|------|----------|
| **命令入口** | `init`、`jobs:sync`、`recommend:run`、`search:run`、`potential:run`、`interaction:run`、`workflow:run`（全流程含同步职位/推荐/搜索/潜在/互动/日报）、`daemon:run`（定时错峰子任务）、`report:daily`。 |
| **搜索↔互动循环** | **`search-interaction:loop`**：同一持久化浏览器会话内，**仅**重复执行「搜索人才 → 互动区」，不含职位同步/推荐/潜在/日报；`searchInteractionLoop.maxRounds` 控制总往返轮数（`0` 表示无限循环直至进程结束）。 |
| **搜索人才** | 进入「搜索人才」、可选保证「未聊过」筛选；按 `manualKeyword` 或岗位生成词搜索；**向下滚动**列表以凑满 `maxCandidatesPerQuery`（默认 20，应对懒加载）；对列表候选人依次：**点打招呼 →「选择沟通职位」（KM 弹窗）→ 发送开场白**；若 `sendResumeRequestAfterOpening` 为 true，**同一轮再发索要简历话术**。搜索列表**不按** `job.excludeKeywords` 做列表级过滤（与「关键词搜到即触达、沉淀简历库」一致）；触达节奏由 `guardrails.minDelayMs/maxDelayMs` 控制。 |
| **推荐 / 潜在** | 读列表、评分、按配置与阈值触达；默认**首个激活岗位**，避免错岗。 |
| **互动区** | 未读优先 + 自上而下兜底扫描会话（上限 `interaction.unreadLimit`）；索要简历、拒绝结束、致谢、转人工；**附件/在线简历**识别与**下载**、本地归档；`llm.enabled` 控制是否走 LLM 结构化决策，否则规则兜底。`automation.autoWorkEnabled=false` 时跳过自动回复与自动下载等「干活」逻辑。 |
| **浏览器与启动** | Playwright + 持久化目录 `data/browser-profile`；**SingletonLock / profile 占用**时：尝试结束占用进程并清理锁文件后重试启动。 |
| **数据与风控** | 候选人状态库、动作日志、互动审计、日报与人工接管/复核列表；`guardrails`：日/小时触达上限、冷却、连续错误停机等。 |

## 3. 核心流程

### 3.1 主线 A：岗位关键字搜索 → 打招呼 → 要简历 → 互动跟进（当前主推）

1. （可选）同步职位中心，保证岗位与话术上下文一致。  
2. **搜索人才**：用配置关键词检索 → 列表触达至多 `maxCandidatesPerQuery` 人 → 每轮打招呼并在允许时**首轮即索要简历**（由 `sendResumeRequestAfterOpening` 等控制）。  
3. **互动页**：候选人回复后按规则/LLM 继续沟通、催简历、识别附件并**下载归档**。  
4. 全量记录状态、日志、人工接管清单。

### 3.2 主线 B：推荐 / 潜在 → 评分 → 触达 → 互动（传统路径）

1. 同步职位中心岗位。  
2. 推荐或潜在页发现候选人并**评分**。  
3. **达阈值**后发送首轮招呼（是否同轮索简历由配置与页面能力决定）。  
4. 互动页读取完整历史对话并做决策：  
   - 候选人积极回复 -> 委婉索简历  
   - 明确不感兴趣 -> 礼貌结束  
   - 识别到附件简历组件 -> 致谢并转人工  
   - 未收到简历前（含敏感/未知问题）-> 统一先索要简历  
5. 全量记录状态、日志、人工接管清单。

## 4. 非功能需求

- 稳定性：单任务失败不拖垮守护进程。
- 可追溯：每次决策可回放（输入摘要、模型输出、执行结果）。
- 安全性：动作白名单、频控、黑名单、敏感词兜底。
- 可维护：配置驱动 + 模块化，避免硬编码分散。

## 4.1 AI 原生决策原则

- 能由 AI 基于上下文稳定判断的场景，不再以关键词规则作为主链路。
- 规则系统只承担安全边界与异常回退，不与 AI 主链路竞争决策权。
- 所有自动动作必须由结构化决策输出驱动，禁止散落式 if-else 关键词决策。

## 5. LLM 决策需求（互动页）

### 5.1 输入

- 双方最近 N 轮完整消息（默认 6~12 条，按实现可调）。
- 候选人状态与岗位上下文。
- 是否出现“查看附件简历/附件简历”组件。
- 风控规则提示（敏感词、拒绝提示词、积极提示词）。

### 5.2 输出

固定 JSON：

- `intent`: `positive | negative | resume_sent | question | sensitive | unknown`
- `action`: `resume_request | closing | ack_and_handover | handover | noop`
- `reply_text`: 文案
- `confidence`: 0~1
- `reason`: 简述依据

### 5.3 执行规则

- 执行层只接受白名单 action。
- `ack_and_handover` 后必须进入人工跟进链路。
- `handover` 仅在已收到简历后才允许触发。
- 重复消息去重，避免同一上下文重复触发。

## 6. 当前版本边界

- 仅支持**智联招聘**（`rd6.zhaopin.com`）页面自动化（Playwright）。
- 推荐/潜在页默认处理**首个激活岗位**，避免错岗归属。
- 互动区已支持**简历附件下载与本地归档**（具体目录见 `storage.resumeDir`）；另保留对话证据与结构化线索。
- **`search-interaction:loop`** 不包含职位同步、推荐、潜在、日报；需要时请单独跑 `workflow:run` 或对应子命令。

## 8. 验收标准

- 能连续运行 24h，互动轮询维持 1 分钟。
- 未读互动可在下一轮内处理，不遗漏明显会话。
- 对“拒绝/已发简历/未发简历索要”三类场景处理正确率可人工抽检通过。
- 状态文件、日报、人工接管列表一致。

## 11. 当前对齐需求（2026-03-27）

### 11.1 目标（第一性原理）

- 核心输入：岗位关键字 + 地域约束（东北）。
- 核心动作：搜索候选人 -> 触达候选人（打招呼/可发首条消息时索要简历）。
- 核心产出：候选人回复后进入互动区，最终拿到并下载简历，沉淀到本地简历知识库。

### 11.2 业务闭环（最小可行）

1. 在人才搜索页使用岗位关键字检索候选人（东北地区优先）。  
2. 每次任务最多触达 20 个候选人（风控硬约束）。  
3. 触达阶段先打招呼；若页面能力允许，直接发送礼貌索简历话术。  
4. 触达完成后结束前链路，不在该阶段做复杂多轮对话。  
5. 候选人有回复后，由互动区既有逻辑接管处理。  
6. 候选人发送简历后，执行下载与归档，进入简历资产沉淀链路。  

### 11.3 当前已实现（与代码一致，2026-03-28）

- **搜索人才**：关键词搜索、列表滚动、`maxCandidatesPerQuery`、打招呼、KM「选择沟通职位」、开场白 + 可配置同轮索要简历；搜索触达策略与推荐页评分策略**分离**（列表不按 `excludeKeywords` 过滤）。
- **`search-interaction:loop`**：搜索与互动在同一浏览器内往返，`maxRounds` 可配；依赖 `automation.autoWorkEnabled` 执行互动侧自动逻辑。
- **推荐 / 潜在 / 互动**：列表采集、评分、触达、互动规则或 LLM、简历下载与归档、会话去重与防护。
- **职位同步**、**日报/人工列表**、**守护进程**定时任务。
- **持久化浏览器 profile** 与 **SingletonLock** 自动恢复。
- **风控**：`guardrails`（日/小时上限、间隔、冷却等）；状态与审计日志。

### 11.4 待实现或增强

- **地域过滤**：`search.northeastOnly` 等可按需强化（城市映射、与列表字段对齐）。
- **失败续跑**：单候选人发送失败时是否跳过并继续下一人的策略可配置化（当前部分路径会中断）。
- **运营看板**：接管量、触达转化等（见 §7 P2）。
- 其它见 §7 迭代清单。

### 11.5 非目标（本轮不做）

- 不新增复杂对话策略（继续复用互动区现有能力）。
- 不做大规模模型策略重构（保持当前规则/LLM架构）。
- 不扩展新的报表系统或管理后台。

### 11.6 验收标准（本轮）

- 可基于岗位关键字稳定检索候选人列表（地域过滤为可选配置）。
- 单次搜索任务触达人数不超过 `maxCandidatesPerQuery`（常与 20 对齐），且触达行为可审计。
- 若配置允许，首轮即可发送索要简历话术；否则至少完成打招呼与选职流程。
- **`search-interaction:loop`** 能完成至少一轮「搜索 → 互动」且浏览器与会话状态正常。
- 候选人回复后，能由互动区既有链路处理；简历下载与归档可追溯。

### 11.7 搜索人才打招呼：已遇到问题与实现结论（2026-03-28）

| 现象 | 原因 | 处理 |
|------|------|------|
| 启动报 `SingletonLock` / profile 已占用 | 上次浏览器未关或异常退出残留锁文件 | 启动前自动结束占用 `SingletonLock` 的进程并清理锁文件后重试 |
| 只打了约 5～6 人而非 20 | 首屏 DOM 仅挂载少量 `.search-resume-item-wrap`，懒加载未触发 | 搜索后向下滚动列表，直至行数达到 `maxCandidatesPerQuery` 或不再增加 |
| 解析 0 条或只处理 1 人 | 曾受 `topContactCount`、列表层 `excludeKeywords` 过滤 | `topContactCount` 为 0 时按 `maxCandidatesPerQuery` 拉满；搜索列表不再按岗位排除词过滤 |
| 「未聊过」日志提示未找到 | 部分页面为自定义筛选项，非标准 `checkbox` | 增加文案「未聊过」检测；筛选仍以页面「未聊过」为准 |
| 选职弹窗后长时间无操作 | 原固定 `sleep(3500)` 等 IM | 改为轮询直至出现聊天输入框（或超时），点确定后等待弹窗关闭 |
| 两人间隔过长 | `guardrails.minDelayMs/maxDelayMs` 过大 | 可按真人节奏改为约 1000～2000ms |
| 日志「第 N 套定位器」 | KM 职位搜索框多套 Playwright 定位器依次尝试 | 仅调试信息，表示第 N 套选择器命中输入框 |
| 弹窗已出现但迟迟不输入职位关键字 | 原先用宽泛正则等待文案、或等职位列表接口 | 已收窄为「选择沟通职位」+ `.km-modal--open`；缩短选项 `waitFor` 单次时长 |

### 11.8 编排命令对照（产品视角）

| 用户目标 | 推荐命令 |
|----------|----------|
| 只做搜索打招呼（单任务） | `search:run` |
| 只做互动与简历处理 | `interaction:run`（需 `autoWorkEnabled=true`） |
| **搜索与互动交替跑、简历库持续沉淀** | **`search-interaction:loop`** |
| 职位同步 + 推荐 + 搜索 + 潜在 + 互动 + 日报一条龙 | `workflow:run` |
| 长期值守、多任务按间隔轮询 | `daemon:run` |
